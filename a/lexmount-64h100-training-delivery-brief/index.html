NVIDIA / NeMo-Gym · 64 H100 training

Lexmount 64 H100 训练交付包

目标不是交付一条看起来能启动的命令，而是让 NVIDIA 在不给我们 SSH 的情况下，独立部署、运行、恢复、评测并审核一组 Lexmount 浏览器训练实验。

Proposed

本页是交付契约草案，不宣称已完成 64 卡实跑。运行入口、版本锁定、失败证据和验收标准会随模型、数据与集群约束确认后冻结。

Short answer

交付一个私有、可复制的实验包：一个总入口完成预检、环境、模型与数据、预处理、分布式训练、阶段评测、Lexmount / 本地 Chrome 对照和诊断归档；支持在可恢复的前提下从 1 卡 smoke 扩展到 8 节点 × 8 H100。

What must work

这不是“启动训练”的脚本，而是一份无人值守的实验运行说明书。

假设 NVIDIA 只拿到我们的私有包、不会给我们 SSH，脚本必须自行完成下面每项工作，并把每项的输入、版本、结果和失败原因写入可审计产物。任何一项缺失，交付都还不完整。

必须能力脚本实际要做什么必须留下的产物
1. 部署环境检查集群前置条件，拉取或构建固定镜像，安装锁定的 CUDA / PyTorch / NeMo / Lexmount SDK；从私有 secrets/lexmount.env 加载 key 和 project_id。镜像 digest、Python lock、SDK 版本、预检报告；密钥不写进日志。
2. 下载模型和数据下载模型权重、tokenizer、训练/评测数据；核对 revision、checksum、许可证和本地缓存命中。assets_manifest.json：来源、revision、哈希、缓存路径、下载状态。
3. 数据预处理固定数据切分、格式转换、过滤/验证、tokenization 或轨迹准备；可重复执行且不会损坏已完成缓存。样本数、过滤原因、split 哈希、预处理配置与数据质量报告。
4. 跑实验与恢复按 8 节点 × 8 H100 启动；支持 Slurm 或 torchrun；保存 checkpoint、dataloader 状态和阶段状态；对临时请求故障有界重试。run manifest、rank 日志、checkpoint 索引、故障分类和恢复记录。
5. 曲线和中间评测记录训练曲线，按固定 step 在固定任务集上评测；避免只在最后一步挑一个结果。loss/reward 曲线、verifier pass rate、任务成功率、阶段评测明细。
6. 对比两种浏览器在相同模型、任务、seed、并发、超时和重试下分别运行 Lexmount 与 local Chrome；检查所有参数的 hash。参数差异表、可比性结论、后端结果与资源成本对照。
7. 规模预检验证 64 张 H100、驱动、NCCL、IB/RDMA、共享存储、网络、容器运行时和磁盘空间。节点/GPU 拓扑、NCCL 诊断、网络/存储检查与明确的 fail-fast 原因。
8. 完整可复现锁定代码 commit、镜像、模型/数据 revision、超参和随机 seed；一次 run 对应一份不可混淆的 manifest。可复现 manifest 与所有版本、哈希、命令行和环境变量白名单。

01 / package

交付目录以“可运行包”组织，而不是一组彼此松散的脚本。

主脚本负责阶段编排、幂等检查和退出状态；支撑文件明确复现所需的配置、依赖、启动后端和诊断逻辑。NVIDIA 只需拿到私有包，在满足前置条件的集群上运行一个命令。

README.md

前置条件、单命令运行、预计产物、分层验收、失败排查与支持边界。

run_64h100.sh

唯一总入口。按顺序执行 preflight、bootstrap、asset、prepare、train、eval、compare 与诊断归档。

Dockerfile + requirements.lock

固定 CUDA / PyTorch / NeMo / Lexmount SDK 版本；避免依赖“集群上刚好有某个环境”。

configs/experiment.yaml

模型、数据、训练超参、评测频率、并发、seed、重试策略和 checkpoint 策略。

configs/cluster_64h100.yaml

8 节点 × 8 GPU、launcher、NCCL/网络与共享路径约定；允许按 NVIDIA 实际调度器覆盖。

scripts/{preflight,bootstrap,download,prepare,train,eval,compare,diagnostics}.*

阶段化实现；每项可独立重试，但不改变总入口的记录与版本语义。

secrets/lexmount.env

只存在于私有交付包。包含 Lexmount key 与 project_id，权限为 0600；Git、Docker 镜像、运行日志和此 Helm 文档仅保留 .example 模板。

02 / parity

比较必须只改变浏览器后端；否则不能把结果称为 Lexmount 优势。

训练和评测都应共用一份 run manifest。若任意强约束不一致，脚本输出参数差异并把本轮比较标记为不可比，而不是静默生成结论。

维度Lexmount本地 Chrome比较规则
保持一致同一模型、数据切分、prompt、训练步数同一模型、数据切分、prompt、训练步数完全相同；由 manifest 和哈希校验
执行参数同一并发、最大 steps、超时、重试、seed同一并发、最大 steps、超时、重试、seed不一致即标记为不可比
唯一变量Lexmount remote browser / SDK训练节点上的 local Chrome runner浏览器后端是唯一有意差异
记录指标成功率、操作延迟、失败/重试、资源占用成功率、操作延迟、失败/重试、资源占用同时记录训练与环境成本，不只比较 reward

03 / observe

最终交付的不只是一个 checkpoint，而是一套能解释结果的运行证据。

训练收益和环境收益必须分开记录。否则即使 Lexmount 的最终 reward 更高，也无法判断提升来自训练、任务完成、浏览器稳定性，还是机器侧资源差异。

训练曲线

global step、loss、reward、learning rate、tokens/s、samples/s、梯度/溢出异常、checkpoint 时间与恢复次数。

任务与 verifier

固定评测集上的任务成功率、verifier pass rate、完成步数、失败类别、每类任务的分项结果。

浏览器后端

浏览器启动时间、页面/操作延迟、超时、会话失败、重试次数、任务级成功率与并发实际值。

系统与集群

每节点 GPU 利用率/显存、CPU/RAM、磁盘 I/O、网络吞吐、NCCL 错误与 GPU 小时。

输出文件用途最低要求
artifacts/metrics/train.jsonl训练曲线与 step 级事件可离线画图，不依赖线上 dashboard。
artifacts/eval/step-*.json中间评测结果与任务级明细相同 eval split、相同 verifier、相同评测参数。
artifacts/metrics/system-*.jsonl节点与 GPU 资源使用至少覆盖训练窗口和评测窗口。
artifacts/comparison_report/两个后端的参数、结果与成本对照先输出可比性，再输出任何优劣结论。

04 / recovery

降级必须可见、可逆、不会偷换实验条件。

Expected recovery

短暂故障可恢复

资产下载具备校验与缓存；浏览器/网络请求做有界重试；训练从最近 checkpoint 恢复；阶段状态写入 run_manifest.json。中断后重跑总入口应跳过已验证阶段。

Hard boundary

不可静默降级

GPU 少于 64、NCCL/IB 不可用、评测集缺失、浏览器参数不一致或凭据失效时，正式 run 必须失败并输出诊断包；可以另开降级 smoke，但不可混入 64 卡结论。

阶段恢复动作保留证据
环境 / 资产版本与 checksum 一致时复用缓存；不一致时停止镜像 digest、wheel lock、模型与数据 revision
训练从最近完整 checkpoint 和 dataloader 状态恢复global step、rank 状态、NCCL/torch 分布式日志
浏览器 / 评测有界重试后写失败分类；不改变评测规则任务级结果、重试次数、超时、backend 版本

05 / acceptance

验收分层：先证明入口正确，再要求 64 H100 的规模运行。

  • Dry run：验证配置、文件、凭据引用与调度命令，不下载或训练。
  • 1 GPU smoke：完成最小数据预处理、一个真实训练 step、一次 Lexmount 与本地 Chrome 评测。
  • 8 H100 node check：验证多卡启动、NCCL、checkpoint 恢复、指标聚合和浏览器并发。
  • 64 H100 run：完成 8 节点 × 8 卡训练，并输出训练曲线、阶段评测、资源指标、版本 manifest 和后端对照报告。
  • 诊断可读：任一门槛失败时，artifacts/diagnostics/ 内存在 GPU、网络、NCCL、环境、阶段状态和最近日志的可打包证据。

06 / confirm

需要先与团队和 NVIDIA 对齐的不是文案，而是这六个执行约束。

训练入口

模型仓库、NeMo/训练框架版本、已有 entrypoint 与预期训练目标。

模型与数据

下载地址、revision、许可证、访问令牌、数据预处理规则和缓存路径。

集群形态

Slurm 还是裸机；节点数、GPU 拓扑、IB/RDMA、共享存储与容器运行时。

网络与权限

是否可访问 PyPI、镜像仓库、模型仓库和 Lexmount；无法访问时由谁提供离线包。

评测定义

固定任务集、verifier、阶段评测频率、成功率与资源效率的最终口径。

比较边界

本地 Chrome runner 的位置和版本；并发、超时、重试、seed 以及不可比判定阈值。
